Understanding TRL

# Understanding TRL, PPO, Value Heads, and TD Errors in LLM RL

This post summarizes a deep dive into TRL (Training Reinforcement Learning) and how PPO-based RLHF works for large language models, focusing on value heads, token-level rewards, and TD error–based credit assignment.
It also explicitly documents the confusions that naturally arise when first studying this system.

# 1. Where TRL Fits in LLM RL

TRL is a training framework, not a model and not a reward system.

It sits between:

  • a Transformer policy model
  • a reward signal (reward model, rules, or external evaluator)
  • a reinforcement learning algorithm (usually PPO)

TRL’s job is to:

  • generate rollouts
  • assign rewards
  • compute advantages
  • update the model safely (KL-regularized PPO)

# 2. Policy Model Structure

The policy model used by TRL has two heads on top of the same transformer backbone:

Transformer
 ├─ LM head     → token probabilities
 └─ Value head  → V(s_t), expected future reward
1
2
3
  • The LM head decides what token to generate
  • The value head predicts how promising the current prefix is

The value head runs in parallel with token generation, not afterward.

# 3. Language Generation as an MDP

TRL treats text generation as a finite-horizon MDP:

RL Concept LLM Equivalent
State Prompt + generated prefix
Action Next token
Episode end EOS or max length
Reward Scalar score from reward model

The episode typically receives reward only at the final token.

# 4. Reward Model: When and How It Is Used

# Key clarification (common confusion)

The reward model is NOT called per token.

Instead:

  1. The full response is generated first.
  2. The reward model is run once on:
    (prompt, full_response)
    1
  3. A single scalar reward R is produced.
  4. TRL assigns:
    r_T = R
r_0...r_{T-1} = 0
    1
    2

The phrase “token-level rewards” refers to how rewards are stored and propagated, not how they are computed.

# 5. Why Token-Level Credit Assignment Is Still Possible

Even with reward only at the last token, PPO can assign credit backward using:

  • a value function
  • temporal-difference (TD) errors
  • generalized advantage estimation (GAE)

This is identical to episodic RL with terminal rewards.

# 6. What the Value Head Represents (“Prefix Promise”)

The value head estimates:

V(s_t) ≈ expected final reward if we continue from this prefix
1

This is called prefix promise.

  • High value → prefix likely leads to a good final response
  • Low value → prefix likely leads to a poor outcome

Prefixes differ in promise even before the reward is known.

# 7. TD Error: Meaning and Interpretation

TD error at token t is:

δ_t = r_t + γ V(s_{t+1}) − V(s_t)
1

# For most tokens (no immediate reward):

δ_t = V(s_{t+1}) − V(s_t)
1

Interpretation:

  • Positive TD error → the token improved the trajectory
  • Negative TD error → the token hurt the trajectory
  • Zero TD error → the token behaved as expected

TD error measures surprise, not goodness.

# 8. Terminal TD Error

At the final token:

δ_T = R − V(s_T)
1

This anchors learning:

  • If reward > expected → positive signal
  • If reward < expected → negative signal

This terminal error is propagated backward to earlier tokens.

# 9. From TD Error to Advantages (GAE)

Advantages are computed as:

A_t = δ_t + γλ δ_{t+1} + γ²λ² δ_{t+2} + ...
1

This:

  • spreads terminal reward backward
  • smooths noise
  • produces token-specific learning signals

Advantages, not rewards, drive PPO updates.

# 10. PPO Update at Token Level

Each token contributes independently to training:

  • Policy loss increases or decreases token probability based on advantage
  • Value loss trains the value head to better predict future reward
  • KL penalty prevents drift from the reference model

Even though reward is terminal, learning is distributed across all tokens.

# 11. TD Error in the “Ideal” Model

In theory:

  • A perfect value function → expected TD error = 0

In practice for LLMs:

  • Mean TD error ≈ 0
  • Variance > 0 (and should remain)

Exactly zero TD error everywhere would indicate:

  • stalled learning
  • over-regularization
  • or a frozen policy

# 12. How TD Error Is Implemented on GPU

There is no special CUDA kernel for TD error.

Implementation uses standard tensor ops:

td_error = rewards + gamma * next_values - values
1
  • Fully vectorized over batch and time
  • Executed as simple CUDA elementwise kernels
  • Computationally negligible compared to transformer forward passes

GAE requires a reverse-time scan:

  • Sequential over time
  • Parallel over batch

# 13. Key Confusions (and Resolutions)

# ❓ Is reward computed per token?

No. It is computed once per full response. And only the last token has nonzero reward. All other token reward is zero.

# ❓ Are token-level rewards fake?

No. They are created via credit assignment.

# ❓ Does the value head run after generation?

No. It runs during generation, at every step.

# ❓ Is higher TD error better?

No. TD error is a prediction error; sign matters.

# ❓ Should TD error go to zero?

Only in expectation, never exactly.

# 14. One-Sentence Mental Model

TRL trains a transformer by learning which prefixes lead to good endings, using a value head to propagate terminal rewards backward through token-level PPO updates.

# 15. Why This Design Works

  • Language has delayed reward
  • Many valid trajectories exist
  • Prefix quality matters more than final token alone

TRL succeeds because:

  • value heads model prefix promise
  • GAE propagates reward backward
  • PPO updates tokens independently
  • KL preserves fluency

# Final Takeaway

Even though RLHF looks mysterious, it is:

standard episodic RL applied carefully to text, with the value head acting as a learned future-reward surface over prefixes.

Once this mental model clicks, TRL becomes conceptually simple — even if the code is large.
