# [1] vLLM Office Hours - Distributed Inference with vLLM - January 23, 2025

link (opens new window)

# 1.Summary of Video: "vLLM Office Hours - Distributed Inference with vLLM"

The video detailed the need for distributed inference to serve large LLMs (like 600B+ parameters) that exceed single-GPU capacity.

# Distributed Techniques:

Technique Goal Mechanism Trade-offs
Tensor Parallelism (TP) Fit model on a single node; improve latency and throughput. Shards weights horizontally across multiple GPUs (SPMD paradigm). Uses AllGather/AllReduce operations. High communication overhead, mitigated by high-speed interconnects (e.g., NVLink).
Pipeline Parallelism (PP) Fit model across multiple nodes. Shards the model by layer across multiple GPUs/nodes. Uses Send/Receive operations. Low communication overhead, but doesn't inherently improve latency and can suffer from GPU idle time ("bubbles").
Combination PP can be used across nodes for low communication, and TP within a node for speed and memory efficiency.

# 2.Key Optimization: Chunk Prefill

Chunk Prefill is critical, especially for PP, and involves breaking large prefill operations into smaller chunks and mixing them with decode batches.

  • Impact of Chunk Size: A small, carefully chosen size is essential for a smooth, bubble-free execution pipeline and maximum throughput.
  • Memory Management: It limits the memory spike from long inputs, allowing the system to safely allocate more KV Cache space, which boosts concurrency.

# 3. vLLM Architecture Components (Scheduler, Executor, Worker)

These components coordinate to manage and execute inference requests in a distributed vLLM environment:

  • Scheduler: Manages the flow of requests, schedules batches for processing, and receives the final results.
  • Executor: The primary distributed manager. It manages the Workers, handles hardware backends (Ray, multi-processing), and issues the coordinated distributed inference commands.
  • Worker: The execution unit associated with a specific accelerator (e.g., GPU). It performs the actual forward pass and inference computation.

# 4. Definition of Chunk Prefill Size

The Chunk Prefill Size parameter dictates the maximum number of tokens from a large input prompt that vLLM will process in a single step (or "chunk").

It serves to:

  1. Enforce Memory Safety: Prevents large, arbitrary inputs from causing a sudden, massive spike in activation memory, thus preventing Out-of-Memory (OOM) errors.
  2. Amortize Cost: Smoothes the workload by breaking the long prefill into smaller, manageable pieces that can be interspersed with decode operations.
  3. Boost Concurrency: By guaranteeing a maximum memory footprint, the system can allocate more space for the KV Cache, leading to higher overall throughput.

# 5. Tensor Parallelism vs. Pipeline Parallelism Comparison

The fundamental difference lies in how the model is split:

Feature Tensor Parallelism (TP) Pipeline Parallelism (PP) / Layer Parallelism
Model Split Intra-Layer Parallelism: Splits computation/weights within individual layers. Inter-Layer Parallelism: Splits the model by layer or group of layers (stages).
Communication Frequent: Communication (All-Reduce/All-Gather) happens after nearly every linear layer. Infrequent: Communication (Send/Receive of activations) happens only at the boundaries between stages.
Best Used For Scaling within a Node (high-speed interconnects like NVLink). Scaling across Nodes (slower interconnects).
Goal Allow a single large layer/tensor to fit and speed up computation. Allow the entire model to fit by distributing its total memory footprint.

# 6. Column and Row Parallelism in LLMs

Tensor Parallelism is implemented using two specialized patterns within the Transformer layers, especially in the Multi-Layer Perceptron (MLP) block:

  1. Column Parallelism (ColumnParallelLinear):
    • Mechanism: Splits the weight matrix by columns.
    • Flow: The input is multiplied by the sharded weight matrix. The output is already sharded across GPUs.
    • Communication: No communication is required immediately after the multiplication (e.g., after the Up-Projection layer).
  2. Row Parallelism (RowParallelLinear):
    • Mechanism: Splits the weight matrix by rows.
    • Flow: The input (which is sharded from the previous Column Parallel step) is multiplied by the sharded weight matrix.
    • Communication: Requires a final All-Reduce operation to sum the partial results from all GPUs and generate the correct output (e.g., after the Down-Projection layer).

LLMs combine these: The Up-Projection uses Column Parallelism (no communication), and the subsequent Down-Projection uses Row Parallelism (single All-Reduce).

This trick is used through all layers.

This minimizes the required synchronization steps.

Major takeaway:

  1. first column parallelism, then row parallelism
  2. prefer tensor parallelism, gpu is parallelized but with heavy communation, this is prefered in intranodes, communicate will happen in each layer As to pipeline parallelism, there might be bubble accross gpus. When gpu0 is done with layer0-3, it passes the intermediate result into gpu1 for layer4-7. Only result from layer3 needs to be transfered.

# [2]. Lowering in VLLM into CUDA Code Generally

# 1. What vLLM Is (and Is Not)

  • vLLM is not a compiler (unlike TVM, XLA, or MLIR-based systems).
  • It is a runtime + kernel library built on top of PyTorch.
  • Core idea:
    • Use PyTorch as the frontend
    • Replace critical paths (attention + KV cache) with custom CUDA/Triton kernels
    • Add a scheduler and memory manager (PagedAttention)

# 2. High-Level Execution Flow

Python model.forward (vLLM)
  ↓
PyTorch ops + custom vLLM ops
  ↓
Torch dispatcher
  ↓
C++ extension (ATen)
  ↓
CUDA / Triton kernels
  ↓
GPU execution

vLLM routes execution to prewritten kernels, not IR-generated CUDA.

# 3. Python Model Level

  • Model definitions resemble standard PyTorch LLMs
  • Key differences:
    • Attention layers are replaced
    • KV cache is externalized
    • Dynamic / ragged batching is supported

# 4. PyTorch Graph & torch.compile

  • vLLM supports:
    • Eager mode
    • torch.compile (Inductor)
  • Limitation:
    • PagedAttention is not lowered by Inductor
    • It remains a custom operator

# 5. Dispatcher Hijacking

vLLM registers custom Torch operators:

TORCH_LIBRARY(vllm, m) {
  m.def("paged_attention(...) -> Tensor");
}
1
2
3

Called from Python as:

torch.ops.vllm.paged_attention(...)
1

This bypasses FX tracing and ATen decomposition.

# 6. C++ Extension Layer

Responsibilities: - Tensor metadata unpacking - Shape/stride computation - Kernel argument preparation

At this point, all compiler IR abstractions are gone.

# 7. CUDA-Level Execution

# (A) Handwritten CUDA Kernels

  • Used for:
    • PagedAttention
    • KV cache gather/scatter
    • Block management
  • Compilation: nvcc → cubin/fatbin → loaded by PyTorch

# (B) Triton Kernels

  • Used for softmax and some fused ops
  • Lowering pipeline: Triton DSL → Triton IR → LLVM IR → PTX → CUDA
  • Only place LLVM is used in vLLM

# 8. GEMM and Linear Layers

  • vLLM does not reimplement GEMM
  • Standard path: PyTorch → ATen → cuBLAS / cuBLASLt → CUDA

# 9. KV Cache & PagedAttention

# Standard Attention

  • Contiguous KV tensors

# vLLM PagedAttention

  • Fixed-size KV blocks
  • Logical-to-physical block tables
  • Indirect memory addressing in kernels

Handwritten CUDA is required due to dynamic memory access patterns.

# 10. Lowering Stages Overview

Stage Tool Role
Python vLLM Model + scheduling
Graph PyTorch Partial graph capture
Dispatch Torch ops Custom op routing
C++ ATen extension Argument preparation
Kernel CUDA / Triton Execution
Hardware GPU SM scheduling

# 11. Key Takeaway

vLLM does not lower models to CUDA via compiler IR. Instead, it selectively bypasses the compiler stack and dispatches execution to highly optimized CUDA and Triton kernels for precise memory and scheduling control.

# 12. Research Directions

  • MLIR-based PagedAttention
  • KV cache compression
  • Runtime scheduling policies
  • Triton attention compatible with Inductor
  • Memory-aware kernel specialization

# [3] How vLLM Lowers Python Code to Direct CUDA Operations

This document explains how vLLM integrates pre-compiled CUDA kernels (written in .cu files) into the Python execution path, focusing on the complete lowering pipeline from Python model code to CUDA device execution.

# Overview

vLLM uses a multi-stage compilation pipeline that transforms Python code into optimized CUDA kernels. For operations written directly in CUDA (.cu files), the process involves:

  1. CUDA Kernel Implementation (C++/CUDA)
  2. C++ Wrapper Function
  3. PyTorch Operator Registration
  4. Python Wrapper Function
  5. Model Layer Integration

# Complete Lowering Path

# 1. CUDA Kernel Implementation (C++/CUDA)

CUDA kernels are written directly in .cu files. These kernels are hand-optimized for specific operations.

# 2. C++ Wrapper Function

A C++ wrapper function handles tensor management, validation, and CUDA kernel launch configuration.

File: csrc/quantization/awq/gemm_kernels.cu

torch::Tensor awq_gemm(torch::Tensor _in_feats, torch::Tensor _kernel,
                       torch::Tensor _scaling_factors, torch::Tensor _zeros,
                       int64_t split_k_iters) {
  int num_in_feats = _in_feats.size(0);
  int num_in_channels = _in_feats.size(1);
  const at::cuda::OptionalCUDAGuard device_guard(device_of(_in_feats));

  auto options = torch::TensorOptions()
                     .dtype(_in_feats.dtype())
                     .device(_in_feats.device());
  at::Tensor _out_feats =
      torch::empty({split_k_iters, num_in_feats, _kernel.size(1) * 8}, options);
  int num_out_feats = _out_feats.size(-2);
  int num_out_channels = _out_feats.size(-1);

  auto in_feats = reinterpret_cast<half*>(_in_feats.data_ptr<at::Half>());
  auto kernel = reinterpret_cast<int*>(_kernel.data_ptr<int>());
  auto out_feats = reinterpret_cast<half*>(_out_feats.data_ptr<at::Half>());
  auto scaling_factors =
      reinterpret_cast<half*>(_scaling_factors.data_ptr<at::Half>());
  auto zeros = reinterpret_cast<int*>(_zeros.data_ptr<int>());
  int group_size = num_in_channels / _scaling_factors.size(0);

  // Validation checks
  if (num_out_channels % 64 != 0)
    throw std::invalid_argument("OC is not multiple of cta_N = 64");
  if (num_out_channels % 8 != 0)
    throw std::invalid_argument("OC is not multiple of pack_num = 8");
  if (group_size % 32 != 0)
    throw std::invalid_argument("Group size should be a multiple of 32");
  if (num_out_channels % group_size != 0)
    throw std::invalid_argument("OC is not multiple of Group size");

  const cudaStream_t stream = at::cuda::getCurrentCUDAStream();
  // Launch CUDA kernel with appropriate grid/block configuration
  // ...
  
  return _out_feats;
}

# 3. PyTorch Operator Registration

The C++ function is registered as a PyTorch operator using TORCH_LIBRARY macros, making it accessible from Python.

File: csrc/torch_bindings.cpp

TORCH_LIBRARY_EXPAND(TORCH_EXTENSION_NAME, ops) {
  // Quantization ops
#ifndef USE_ROCM
  // Quantized GEMM for AWQ.
  ops.def(
      "awq_gemm(Tensor _in_feats, Tensor _kernel, Tensor _scaling_factors, "
      "Tensor _zeros, SymInt split_k_iters) -> Tensor");
  ops.impl("awq_gemm", torch::kCUDA, &awq_gemm);

  // Dequantization for AWQ.
  ops.def(
      "awq_dequantize(Tensor _kernel, Tensor _scaling_factors, "
      "Tensor _zeros, SymInt split_k_iters, int thx, int thy) -> Tensor");
  ops.impl("awq_dequantize", torch::kCUDA, &awq_dequantize);
#endif
}

def awq_gemm(
    input: torch.Tensor,
    qweight: torch.Tensor,
    scales: torch.Tensor,
    qzeros: torch.Tensor,
    split_k_iters: int,
) -> torch.Tensor:
    if envs.VLLM_USE_TRITON_AWQ:

# Compilation Process

# Build-Time Compilation

  1. CMake Configuration

    • CMakeLists.txt (CMakeLists.txt) configures the build
    • CUDA files are identified and added to compilation targets

  2. NVCC Compilation

    • NVCC compiles .cu files:
      • CUDA code → PTX (Parallel Thread Execution)
      • PTX → Device-specific binary code
    • C++ code is compiled with standard C++ compiler

  3. Linking

    • Object files are linked into a Python extension module
    • PyTorch C++ API is linked
    • CUDA runtime libraries are linked

  4. Python Extension Module

    • Result: A shared library (.so on Linux, .pyd on Windows)
    • Loaded when vLLM imports _custom_ops

# Runtime Loading

When vLLM is imported:

# vLLM initialization loads the compiled extension
import vllm._custom_ops  # This loads the compiled CUDA kernels

# Integration with torch.compile

When torch.compile is used:

  1. Custom Op Handling

    • Custom ops can be marked as opaque to Dynamo (not traced)
    • Or they can be traced and treated as atomic operations

  2. Compilation Modes

    • Custom ops enabled: Operations call CUDA kernels directly
    • Custom ops disabled: Operations fall back to forward_native() PyTorch implementation

  3. Configuration

    • Controlled via CompilationConfig.custom_ops
    • Example: custom_ops=["all", "-awq_gemm"] enables all except AWQ GEMM

# Benefits of This Approach

  1. Performance: Hand-optimized CUDA kernels provide maximum performance for specific operations
  2. Flexibility: Can be called from Python like any PyTorch operation
  3. Compatibility: Works seamlessly with torch.compile when custom ops are enabled
  4. Maintainability: Clear separation between Python interface and CUDA implementation
  5. Extensibility: Easy to add new CUDA kernels following the same pattern

# Summary

The lowering path for direct CUDA operations in vLLM follows this pattern:

  1. CUDA kernels are written in .cu files (csrc/ directory)
  2. C++ wrappers handle tensor management and kernel launching
  3. PyTorch registration (csrc/torch_bindings.cpp) makes kernels callable from Python
  4. Python wrappers (vllm/_custom_ops.py) provide clean interfaces
  5. Model layers (vllm/model_executor/layers/) call the Python wrappers
