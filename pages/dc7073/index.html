# 📄 The Unreasonable Ineffectiveness of the Deeper Layers: Summary

# 2. [176 2024] The Unreasonale Ineffectiveness of The Deeper Layers

# 💡 Core Discovery

The paper investigates how knowledge is stored in Large Language Models (LLMs) via layer pruning and discovers a surprising robustness in models like Llama-2-70B, which can have up to roughly half of their layers removed with minimal performance degradation on certain downstream tasks.

# 🛠️ Method: Similarity-Informed Layer Pruning

The authors developed a structured pruning strategy guided by the transformer's residual structure and representation similarity.

# 1. Pruning Strategy

The core algorithm for removing $n$ layers involves calculating the angular distance ($d$) between the input to layer $l$ and the output (input to layer $l+n$) of the block of layers.

  • Optimal Selection: The block of layers is chosen by finding the starting layer ($l^*$) that minimizes this angular distance. A small angular distance suggests the representations did not change much across that block, making those layers the most redundant and thus safest to remove.
  • Simpler Heuristic: An even simpler, highly effective strategy was found: remove the deepest layers (excluding the final layer before the head). This is inspired by the finding that deeper layers are generally more similar to their neighbors than shallow layers.

# 2. Healing Step (QLoRA Fine-tuning)

After pruning, a crucial step is performed to "heal" the damage caused by the sudden network mismatch.

  • QLoRA: The healing is done using QLoRA (Quantized Low-Rank Adaptation), a Parameter-Efficient Fine-Tuning (PEFT) method. This allows the entire experiment to be run efficiently on a single 40GB A100 GPU.
  • Effect: Healing significantly restores the model's next-token prediction ability (autoregressive loss/perplexity) to near-unpruned levels.

# 📊 Key Results and Conclusions

# 1. Robustness and Decoupling

  • QA Tasks are Robust: Accuracy on knowledge-intensive Question-Answering (QA) benchmarks like MMLU and BoolQ remains high until a sharp, model-dependent transition point (e.g., 45%-55% for Llama-2 family models).
  • Loss Decoupling: With healing, the C4 validation loss (perplexity) shows continuity and a slow, linear increase through the region where QA accuracy sharply collapses. This decoupling shows that basic language modeling ability can be restored, but the higher-level QA function may be lost at a critical threshold.

# 2. The Role of Deep Layers

The authors tested tasks beyond QA to determine the function of the deeper layers, leading to the main conclusion on layer roles.

Layer Type Critical Function Observed Behavior with Pruning
Shallow Layers Storing knowledge and retrieving information. Pruning is highly damaging (hence the strategy targets deep layers).
Deeper Layers Higher-level computations such as reasoning. Performance on tasks like GSM8k (math reasoning) and HellaSwag (common sense reasoning) shows immediate degradation with any amount of pruning.

Conclusion: The robustness to deep layer removal for QA tasks implies that current pretraining methods may not be properly leveraging the parameters in the deeper layers for knowledge storage, though these deep layers are essential for complex reasoning.

# 4. [43 2024] Think: Thinner Key Cache by Query-Driven Pruning

Key takeaway

They found only certain channels have significant magnitudes in the key cache, whereas the value cache lacks obvious patterns.

Their straightforward assumption is that only key values with largr magnitude should be preserved and small value can be pruned.

Then they select only top N channels with largest magnitue and the N should min the difference between original and pruned attention weights.

# 5. [3 2025] OrthoRank: Token Selection via Sink Token Orthogonality for Efficient LLM inference

Key takeaway

They found that as to the deeper layers, the cosine similarity between sink tokens and other tokens increase.

Thus, they select the top k tokens that are least similiar to the sink token and reserve the top k tokens and sink tokens.

# 7. [2 2025] USTAFAR: Promoting Unstructured Sparsity for KV Cache Pruning in LLM Inference

Pruning Direction refers to the axis along which sparsity is induced when selecting elements for removal. Since both the Key and Value caches are represented as matrices with dimensions [tokens × channels], we consider two primary pruning directions: per-channel pruning, which determines target sparsity across each channel (i.e., across tokens for each channel), and per-token pruning, which determines target sparsity across each token’s cache (i.e., across model dimensions for each token). Output-Awareness refers to the use of a scoring metric that serves as a proxy for estimating each element’s contribution to the operation’s output. Commonly employed in LLM weight pruning [38] and structured KV cache pruning [44], this technique involves computing a score for each pruning unit such as a channel or an element by taking the product of the corresponding element with its associated input.

The findings establish that a simple per-token magnitude-based pruning strategy allows for much higher sparsity levels (up to 70%) while maintaining significantly better model accuracy.

# 1. Key (K) Cache: Per-Token Magnitude

Direction: Per-token. This determines the target sparsity across each token's cache (across model dimensions for each token).

Method: Magnitude-based.

Why: Key caches contain "outlier" channels with very high magnitudes. Per-token pruning effectively captures these specific outlier elements within a token's vector.

Comparison: The authors tested an "output-aware" version for the Key cache but found that simple magnitude-based pruning was nearly as effective and much simpler to implement.

# 2. Value (V) Cache: Per-Token Magnitude (Inherently Output-Aware)

Direction: Per-token.

Method: Magnitude-based.

Why: Value caches have a more uniform distribution of activations, making them hard to prune using structured (channel-wise) methods without losing accuracy.

The "Output-Aware" Twist: In the Value cache, per-token magnitude pruning is mathematically identical to output-aware pruning. In attention, every element of a token's Value cache is multiplied by the same scalar attention score for that token. Therefore, an element's impact on the final output is directly proportionate to its magnitude.
