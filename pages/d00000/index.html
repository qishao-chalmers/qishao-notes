# Mathematical Formulas for Llama Architecture

Written by gemini

This document contains the mathematical definitions for the core components of the Llama (1, 2, and 3) architecture. These formulas are critical for implementing custom CUDA kernels.

# 1. RMSNorm (Root Mean Square Normalization)

Llama uses RMSNorm instead of standard LayerNorm. It normalizes activations based on their magnitude without centering (subtracting the mean).

The Input: A vector $x$ of dimension $d$ (the hidden size). The Parameters: A learnable weight vector $\gamma$ (gamma) of dimension $d$.

Formula:

Given an input vector $x$ of dimension $d$:

$$ \text{RMS}(x) = \sqrt{\frac{1}{d} \sum_{i=1}^{d} x_i^2 + \epsilon} $$

$$ \bar{x}_i = \frac{x_i}{\text{RMS}(x)} \cdot \gamma_i $$

  • $\epsilon$: Small constant for numerical stability (typically $1e-6$).
  • $\gamma$: Learnable weight parameter of dimension $d$.

CUDA Implementation Strategy:

  • One Block per Row:
    Assign one CUDA thread block to process one row (token) of the input.
  • Warp Shuffle Reduction:
    The trickiest part is summing the squares efficiently. Do not use global memory atomic adds. Instead, use __shfl_down_sync (warp shuffle functions) to sum values across threads within a warp, and then use shared memory to sum across warps.
  • Fusing:
    Perform the division and the multiplication by the weight ($g_i$) in the same kernel to minimize memory reads/writes.

# 2. RoPE (Rotary Positional Embeddings)

Rotary embeddings are applied to the Query ($Q$) and Key ($K$) vectors before attention. They rotate pairs of elements in the vector space.

The Input: A vector $x$ (a row from $Q$ or $K$).

The Logic: We split the vector into pairs $(x_{2i}, x_{2i+1})$. For each pair, we rotate them by an angle $\theta_i$ that depends on their position in the vector.

Formula (Scalar/Component view):

For a token at position $m$, and a head dimension index $i \in [0, d/2-1]$, we process pairs of elements $(x_{2i}, x_{2i+1})$:

$$ \theta_i = \text{base}^{-2i/d} $$

$$ \begin{aligned} x'{2i} &= x{2i} \cos(m \theta_i) - x_{2i+1} \sin(m \theta_i) \ x'{2i+1} &= x{2i} \sin(m \theta_i) + x_{2i+1} \cos(m \theta_i) \end{aligned} $$

Constants:

  • Llama 2 Base: $10,000.0$
  • Llama 3 Base: $500,000.0$

CUDA Implementation Strategy:

  • Pair-wise Threading:
    Each CUDA thread should handle one pair of elements (e.g., indices $2i$ and $2i+1$).
  • Precomputed Cos/Sin Tables:
    Do not compute cos() and sin() inside the kernel every time. Precompute these values (based on the sequence length and embedding dimension) on the CPU, upload them to the GPU once, and read them in the kernel.
  • In-place Operation:
    You can often modify the query ($Q$) and key ($K$) cache in place to save memory.

# 3. Attention with GQA (Grouped Query Attention)

Llama 2 (70B) and Llama 3 use GQA, where multiple Query heads share a single Key/Value head.

Dimensions:

  • $H_q$: Number of Query heads.
  • $H_{kv}$: Number of KV heads (where $H_q$ is a multiple of $H_{kv}$).

Formula:

$$ \text{Attention}(Q, K, V) = \text{Softmax}\left( \frac{Q K^T}{\sqrt{d_{head}}} + M \right) V $$

  • $M$: Causal mask ( $-\infty$ if $col > row$, else $0$).
  • Note: In implementation, $K$ and $V$ are effectively "broadcast" or repeated $(H_q / H_{kv})$ times to match the number of Query heads during computation.

# 4. SwiGLU (Feed-Forward Network)

The FFN layer uses the Swish-Gated Linear Unit (SwiGLU) instead of a standard ReLU or GeLU MLP. It involves three linear projections ($W_{gate}, W_{up}, W_{down}$).

Formula:

$$ \text{SiLU}(x) = x \cdot \sigma(x) = \frac{x}{1 + e^{-x}} $$

$$ \text{FFN}(x) = \text{SwiGLU}(x) \cdot W_{down} $$

Expanded:

$$ \text{FFN}(x) = \left( \text{SiLU}(x W_{gate}) \odot (x W_{up}) \right) W_{down} $$

  • $\odot$: Element-wise multiplication (Hadamard product).

CUDA Implementation Strategy:

  • Simple Element-wise Kernel:
    This is the easiest kernel to write. You launch enough threads to cover the dimension ($N$).
  • Vectorized Memory Access:
    To make it fast, use vectorized loads/stores (e.g., float4). Instead of reading 1 float at a time, read 4 floats per thread. This drastically reduces "memory transaction pressure."
  • Optimization:
    Since SwiGLU usually follows a GEMM, advanced implementations try to "fuse" this into the end of the GEMM kernel, but since you are using cuBLAS, you will have to run it as a separate kernel immediately after the cuBLAS call.

# 5. Softmax (Logits)

Used at the final layer to convert logits into probabilities.

The Input: Final embedding vector $x$ of size $d$.

The Weight: Output matrix $W_{out}$ (un-embedding matrix).

Warning:
The vocab size is huge (128,000). A standard softmax kernel might overflow shared memory if not careful. You will likely need a multi-block reduction strategy in CUDA to handle this summation safely.

Formula:

Given a vector of logits $z$ of size $V$ (vocab size):

$$ p_i = \frac{e^{z_i - \text{max}(z)}}{\sum_{j=1}^{V} e^{z_j - \text{max}(z)}} $$

  • Max Trick: Subtracting $\text{max}(z)$ from all elements before exponentiation ensures numerical stability and prevents overflow.

# 6. Embedding lookup

Turn token IDs (integers like 4521) into vectors.

A massive lookup table of size [VocabSize, Dim].

CUDA Implementation:

  • Naive:
    A simple kernel where each thread reads one row from the embedding matrix based on the input token ID and copies it to the working memory.

# 7. Sampling Strategies (Top-K / Top-P)

Once you have probabilities, you need to pick a token.

Where to run it: For single-user inference, it is often faster to copy the logits (128k floats) to the CPU and perform Top-K/Top-P sorting there, because sorting on GPU is complex to implement efficiently.

CUDA Implementation:
If you insist on GPU sampling (for high batch sizes), you will need to use Cub (Cuda Unbound) library primitives like DeviceRadixSort. Writing a fast parallel sort from scratch is an entire project on its own.
