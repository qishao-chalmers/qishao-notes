LLM Speculative Decoding

  1. Accelerating Large Language Model Decoding with Speculative Sampling

# 1. Speculative Sampling: Accelerating Large Language Model Decoding

Speculative Sampling (SpS) is an algorithm designed to accelerate transformer decoding by enabling the generation of multiple tokens from each transformer call. It achieves this without modifying the target model or compromising sample quality.

# 1. The Core Problem: Memory Bandwidth

In standard Auto-regressive Sampling (ArS), generating text is memory bandwidth bound.

  • Bottleneck: The time to generate a single token is proportional to the model size divided by the memory bandwidth.
  • Inefficiency: A model call only generates a single token for every sequence, meaning powerful hardware cannot be fully utilized during the serial generation process.

# 2. The Solution: Speculative Sampling

SpS relies on the observation that the latency of parallel scoring of short continuations (generated by a faster draft model) is comparable to sampling a single token from the larger target model.

# The Algorithm

The process employs two models:

  1. Draft Model ($p$): A smaller, faster auto-regressive model (e.g., 4B parameters).
  2. Target Model ($q$): The larger, more powerful model (e.g., 70B parameters).
# Step 1: Drafting

The draft model generates a short sequence of $K$ tokens (the "lookahead").

# Step 2: Parallel Scoring

The target model scores the draft tokens in parallel. It computes $K+1$ sets of logits from the drafts in a single pass.

# Step 3: Verification (Modified Rejection Sampling)

The algorithm uses a modified rejection sampling scheme to accept a subset of the $K$ draft tokens from left to right, recovering the distribution of the target model.

# 3. Mathematical Verification

To guarantee that the output distribution matches the target model exactly (lossless), SpS uses specific acceptance and resampling rules.

# Acceptance Criterion

For a draft token $\tilde{x}_{n+1}$ at position $n+1$, we accept it with probability:

$$ \text{Acceptance Prob} = \min\left(1, \frac{q(\tilde{x}{n+1}|x_1,...,x_n)}{p(\tilde{x}{n+1}|x_1,...,x_n)}\right) $$

  • $q(x)$ is the probability according to the target model.
  • $p(x)$ is the probability according to the draft model.

# Resampling (If Rejected)

If a token is rejected, we stop verifying subsequent draft tokens. We resample a new token $x_{n+1}$ from the residual distribution:

$$ x_{n+1} \sim (q(x|...) - p(x|...))_+ $$

Where $(f(x))_+$ denotes normalizing the positive difference: $\frac{\max(0, f(x))}{\sum_x \max(0, f(x))}$.

This ensures the final distribution $P(X)$ equals the target distribution $q(X)$.

# 4. Performance & Results

The method was benchmarked using Chinchilla (70B) as the target model and a 4B parameter draft model.

# Speedup

  • Overall: Achieved a 2-2.5x decoding speedup in a distributed setup.
  • HumanEval (Code): ~2.46x speedup. Code tasks often allow for larger speedups because they contain common sub-sequences (e.g., for i in range) that are easier for the draft model to guess.
  • XSum (Summarization): ~2x speedup.

# Trade-offs (The Lookahead $K$)

  • Latency vs. Length: As the lookahead $K$ increases, the number of draft model calls increases linearly, which increases the total loop time.
  • Diminishing Returns: The acceptance rate decreases as $K$ increases because later tokens depend on the acceptance of previous ones.
  • Optimal $K$: For XSum, latency was minimized at $K=3$.

# 5. Key Advantages

  1. No Retraining: Does not require modifications to the target model's parameters or architecture.
  2. Provably Lossless: The sample distribution is provably the same as the target model up to hardware numerics.
  3. Efficiency: It allows generating multiple tokens per target model call, often exceeding the idealized memory bandwidth ceiling of auto-regressive sampling.

# Internals of Speculative Decoding: Parallel Scoring Explained

Yes, your intuition is spot on. You have correctly identified exactly how the target model "cheats" the autoregressive limit to score multiple tokens at once.

To clarify your specific questions:

# 1. Do they work in parallel?

No, they work sequentially (Relay Race style). The draft and target models do not run at the exact same time. It is a relay race:

  1. First: The Draft Model runs (generating $K$ tokens one by one).
  2. Second: The Target Model runs (processing all $K$ tokens at once).

If they ran in parallel, the Target Model wouldn't know what to score because the Draft tokens haven't been created yet.

# 2. How does the Target Model score a series of tokens?

You asked: "Does this mean the target model will concatenate the prompt with the generated series of tokens, and in one forward pass... generate the logits?"

Yes. This is exactly correct.

The Target Model switches from "Generation Mode" (one by one) to "Teacher Forcing Mode" (processing a batch). This is the same mode used during Training.

# The Mechanism: "Causal Masking"

You might know that LLMs generate tokens one by one during inference ($A \rightarrow B \rightarrow C$). However, during training, we feed the entire sentence "A B C" into the model at once, and we want it to predict "B C D" simultaneously.

The Transformer architecture is designed to do exactly this using a Causal Mask:

  • The model receives inputs $[P, \tilde{x}_1, \tilde{x}_2, \tilde{x}_3]$ (Prompt + 3 draft tokens).
  • It runs one single forward pass.
  • Inside the Attention layers, a mask ensures:
    • Position 1 (Prompt) can only see itself. $\rightarrow$ Predicts $\tilde{x}_1$.
    • Position 2 ($\tilde{x}_1$) can see Prompt and $\tilde{x}_1$. $\rightarrow$ Predicts $\tilde{x}_2$.
    • Position 3 ($\tilde{x}_2$) can see Prompt, $\tilde{x}_1$, and $\tilde{x}_2$. $\rightarrow$ Predicts $\tilde{x}_3$.

Because of this mask, the model can calculate the probability for every token in the sequence simultaneously without "cheating" (seeing the future).

# Mathematical Guarantee: Rejection Sampling in Speculative Decoding

To ensure that the final output of Speculative Decoding is lossless (mathematically identical to the Target Model's distribution), a modified rejection sampling scheme is used. This prevents the faster draft model from biasing the results or over-representing certain tokens.

# 1. The Core Objective

We use Rejection Sampling to achieve a balance between the speed of the Draft Model and the intelligence of the Target Model. The goal is to ensure the final probability of any token $x$ appearing in the output is exactly equal to the target distribution $q(x)$.

# 2. The Acceptance Formula

For a draft token $x$, we calculate the probability of acceptance based on the ratio of the target distribution to the draft distribution:

$$ \text{Chance to Accept} = \min\left(1, \frac{P_{\text{target}}(x)}{P_{\text{draft}}(x)}\right) $$

# Step-by-Step Example

Imagine the following scenario for a specific token:

  • Token $x$: "apple"
  • $P_{\text{draft}}(\text{"apple"})$: $0.40$ (40% confidence from the small model)
  • $P_{\text{target}}(\text{"apple"})$: $0.10$ (10% confidence from the big model)

# The Calculation:

$$ \text{Chance to Accept} = \frac{0.10}{0.40} = 0.25 , (25%) $$

# 3. The "Virtual Die" Mechanism

The system generates a uniform random number $r \sim U[0, 1]$ to decide the fate of the draft token:

  • If $r < 0.25$: The token is ACCEPTED.
    • Result: We keep "apple" and move to verify the next token. Even though the Target Model had low confidence, the token is kept to maintain statistical parity.
  • If $r \ge 0.25$: The token is REJECTED.
    • Result: The system discards "apple" and all subsequent draft tokens in that loop.

# 4. Why This Works (Statistical Proof)

If we repeat this generation process 1,000 times:

  1. The Draft Model proposes "apple" 400 times ($40%$ of 1,000).
  2. The Rejection Sampler accepts it only $25%$ of the time it is proposed.
  3. $400 \times 0.25 = 100$.
  4. Final Result: "apple" appears exactly 100 times ($10%$ of the total 1,000 samples).

This exactly matches the Target Model's original desire of 10% probability, proving the method is lossless.

# 5. Handling Rejections

When a token is rejected, we do not simply fail. The Target Model provides a "correction" by sampling from the residual distribution:

$$ P_{\text{new}}(x) = \frac{\max(0, P_{\text{target}}(x) - P_{\text{draft}}(x))}{\sum \max(0, P_{\text{target}}(x) - P_{\text{draft}}(x))} $$

This ensures that even when the draft model is wrong, the very next token generated is guaranteed to follow the Target Model's true distribution.

# Summary of the Flow

Step Action Mode Description
1 Drafting Serial The small model generates $\tilde{x}_1$, then $\tilde{x}_2$, then $\tilde{x}_3$.
2 Concatenation N/A We create a new input tensor: [Prompt, $\tilde{x}_1, \tilde{x}_2, \tilde{x}_3].
3 Verification Parallel The Big Target Model runs one forward pass on this tensor. It outputs a tensor of logits for $K+1$ tokens.
4 Extraction N/A We extract the logits corresponding to the positions of the draft tokens.
5 Comparison N/A We compare these Target Logits against the Draft Logits using the rejection sampling math.

# Why is this faster?

Even though the Target Model is doing "more math" (processing multiple tokens instead of 1), it takes roughly the same amount of time as processing 1 token.

  • Reason: Transformer sampling is highly memory bandwidth bound.
  • Loading the huge weights from memory takes most of the time.
  • For small $K$, the dense matrix multiplies and attention mechanisms remain memory bound, meaning scoring the continuation takes a similar amount of time to sampling a single token.

So, you are paying the "cost" of loading the big model weights once, but getting the "value" of checking multiple tokens.
