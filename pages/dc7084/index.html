# 10. Quartet: Native FP4 Training Can Be Optimal for LLMs – Summary

Summarized by chatgpt

# Objective

Enable fully end-to-end 4-bit FP (MXFP4) training of large language models (LLMs) with minimal accuracy loss, maximizing compute and energy efficiency on NVIDIA Blackwell GPUs.

# Key Contributions

  1. FP4 LLM Training Analysis

    • First systematic study of hardware-supported MXFP4 training.
    • Introduces metrics:
      • effN (parameter efficiency) → forward-pass quantization effect.
      • effD (data efficiency) → backward-pass gradient bias effect.

  2. Quartet Algorithm

    • Forward pass: Hadamard + QuEST → minimizes activation MSE.
    • Backward pass: Round-to-nearest (RTN) → maintains unbiased gradient estimates.
    • Fully low-precision GEMMs for linear layers.

  3. Scaling-Law Framework

    • Quantizes forward and backward separately:
      L(N, D, P_forward, P_backward) = f(N * effN(P_forward), D * effD(P_backward))
      1
    • Predicts optimal precision for accuracy vs runtime.

  4. Efficient GPU Implementation

    • Fuses Hadamard transform, quantization, mask generation, and GEMM.
    • Optimized memory layout for MXFP4 scales and masks.
    • Achieves up to:
      • 2× speedup over FP8 GEMMs
      • 1.6× end-to-end training speedup on RTX 5090

  5. Experimental Validation

    • Llama-family models: 30M → 7B parameters on C4 dataset.
    • Outperforms prior FP4/INT4 methods (LUQ, HALO, Jetfire, LSS) in:
      • Validation loss
      • Parameter and data efficiency
    • Stable training for large-scale LLMs.
    • Post-training MXFP4 quantization reduces perplexity vs baseline (QuaRot).

# Insights

# 1. Forward vs Backward Separation

  • Forward pass affects activations and parameter efficiency (effN) → directly impacts inference speed.
  • Backward pass affects gradient correctness (effD) → biased gradients require more data or hurt convergence.
  • Separate treatment allows best-of-both-worlds precision: QuEST forward, RTN backward.

# 2. Gradient Bias

  • Low-precision backward quantization can systematically alter gradient magnitude/direction.
  • Causes slower convergence or requires more training data.
  • Quartet addresses this with RTN + Hadamard-based alignment.

# 3. Practical Optimality

  • MXFP4 training can be “optimal”:
    • Slight accuracy degradation is compensated by high throughput.
    • Full FP4 training is faster than FP8 or FP16 while maintaining comparable accuracy.

# 4. GPU-Aware Implementation

  • Forward and backward GEMMs in MXFP4, with fused operations for efficiency.
  • Hadamard transforms used to decorrelate inputs/gradients and reduce quantization error.
  • Masking ensures low-precision clipping does not propagate bias.

Quantization takes such an important ratio of total time.

# Takeaway

Quartet shows that LLMs can be trained fully in 4-bit FP precision with near-lossless accuracy if forward and backward passes are handled separately, gradient bias is managed, and GPU kernels are highly optimized. MXFP4 training is both practical and faster than higher-precision alternatives.

# 11. 🚀 [5 2025] LiquidGEMM: Hardware-Efficient W4A8 GEMM Kernel for High-Performance LLM Serving 👍 👍 👍

This is a paper I would like to reproduce

# 1. Introduction

This paper addresses a practical performance bottleneck in W4A8 (4-bit weights, 8-bit activations) quantization for LLMs, which is widely used for efficient inference.

The key insight is that while W4A8 quantization theoretically reduces memory usage and improves throughput, existing W4A8 GEMM kernels underperform in practice, often being slower than W8A8 or even FP16 due to inefficient dequantization on CUDA cores before Tensor Core MMA operations.

The authors propose LiquidGEMM, a hardware-efficient W4A8 GEMM kernel that introduces two co-designed innovations:

  • LiquidQuant (LQQ) – A safe, hardware-efficient quantization/dequantization algorithm.
  • Implicit Fine-Grained Pipeline (ImFP) – A pipeline design that maximizes overlap between weight loading, dequantization, and MMA across GPU warps without software synchronization.

Experimental results demonstrate up to 2.90× GEMM kernel speedup and up to 4.94× end-to-end system speedup over state-of-the-art W4A8 kernels.

# 2. Background & Motivation

# 2.1 W4A8 Quantization

Quantizes FP16/FP32 weights to 4-bit integers and activations to 8-bit integers.

Reduces memory footprint and improves throughput on Tensor Cores.

Problem: Unlike W8A8, W4A8 requires dequantization on CUDA cores before MMA, which is costly because CUDA cores are slower than Tensor Cores.

# 2.2 GEMM on GPUs

GEMM (General Matrix Multiplication) is the main computational bottleneck in LLM inference.

Tensor Cores accelerate matrix multiply-accumulate (MMA) operations but only support symmetric precisions efficiently.

For asymmetric W4A8, weight dequantization is performed in the main loop, leading to a bottleneck (Figure 3, page 4).

# 2.3 Performance Gap

Benchmarks on LLaMA2-7B and Mixtral-8×7B reveal that W4A8 is:

  • Competitive at small batch sizes.
  • Up to 2× slower than W8A8 at larger batch sizes.

Dequantization overhead dominates compute in large-batch and compute-bound scenarios (highlighted on pages 4–5).

Roofline analysis predicted W4A8 should be faster, but practical implementation falls short due to CUDA Core limitations and overflow-handling overhead.

Cost Model:

Without dequantization overhead:

  • W4A8 and W8A8 should be similar in compute-bound scenarios because both use INT8 MMA on Tensor Cores.
  • W4A8 should even outperform W8A8 in memory-bound scenarios, since it loads half the weight data (4-bit vs 8-bit) → lower memory load (T_LD).

Problem:

  • W4A8 requires dequantization on CUDA cores (T_DQ) before Tensor Core MMA.
  • CUDA cores are much slower than Tensor Cores, and dequantization requires multiple instructions per element (especially to handle overflows).
  • This adds significant overhead, which dominates the main loop at larger batch sizes.

Effect on performance curves:

  • T_DQ “shifts” the expected W4A8 performance down.
  • W4A8 is similar to W8A8 in memory-bound scenarios, and up to 2× slower in compute-bound cases (page 5).

Takeaway: The bottleneck isn’t the Tensor Core MMA, but CUDA-core dequantization.

# 2.4 Why increasing batch size doesn’t solve the problem

One might think: “If I increase batch size M, the overhead T_DQ can be amortized across more tiles.”

  • Limitation: Tile size (M_t) is constrained by shared memory and the M_t determinnes the arithmetic instensity.

Extended Question, why it is bounded by M??

This means arithmetic intensity is bounded, and T_DQ cannot be effectively hidden, even at large batch sizes.

Result: the performance gap between theory and practice remains.

# 2.5 Design Implications for Efficient W4A8 GEMM

The analysis suggests two critical design principles:

Pipeline across heterogeneous units:

  • Weight loading (TMA), dequantization (CUDA cores), and MMA (Tensor Cores) must be fully pipelined.
  • Avoids serialization due to dequantization.

Hardware-efficient dequantization:

  • The instruction cost per dequantized element (α) must be low enough to overlap with memory load or MMA.

On H100:

  • Memory-bound threshold: α ≤ 5.07 instructions per element.
  • Compute-bound threshold (M=150): α ≤ 5.05 instructions per element.
  • Additional tasks (address calculation, index updates) further increase the CUDA core load, reinforcing the need for optimized dequantization.

# 3. LiquidGEMM Design

# 3.1 LiquidQuant (LQQ) – Hardware-Aware Quantization

Two-level quantization:

  • FP16 → INT8 (per-channel, protective range [-119, 119]).
  • INT8 → UINT4 (shift into unsigned domain to avoid overflow). Dequantization formula:
  • Uses multiply-add (IMAD) and XOR instructions.
  • Fully avoids overflow in UINT8 domain (Figure 8, page 8).
  • Four elements dequantized with only two hardware instructions, reducing CUDA Core load.

# 3.2 Implicit Fine-Grained Pipeline (ImFP)

Goal: overlap weight loading → dequantization → MMA efficiently. Compare to ExCP (explicit coarse-grained pipeline):

  • ExCP assigns separate warp groups for dequantization and MMA → introduces pipeline bubbles and extra memory transfers.

ImFP:

  • Single-producer (Load WG) → multiple-consumer (Compute WGs).
  • Each Compute WG performs both dequantization and MMA, avoiding intermediate writes to shared memory.
  • Achieves implicit parallelism across warp groups (Figure 6, page 6).

**This part is to eliminate the complex of weight loading via TMA, dequantization via CUDA Cores,and MMA via Tensor Cores.

# 3.3 Memory Layout Optimizations

Dual-MMA packed layout (Figure 7, page 7):

  • Packs data for two consecutive MMA operations per thread.
  • Aligns memory for 32 UINT4 elements per thread → reduces load instructions and shared memory conflicts.
  • Improves utilization of Tensor Cores and shared memory bandwidth.

# 4. Key Contributions

  • Performance Analysis: Pinpointed dequantization as the main W4A8 bottleneck. LiquidQuant: Overflow-safe, hardware-efficient W4A8 quantization/dequantization.
  • ImFP Pipeline: Implicit fine-grained pipelining maximizes GPU subsystem utilization.
  • Memory Layout Optimization: Dual-MMA layout reduces load overhead and maximizes bandwidth. System Integration: End-to-end LLM serving (LiquidServe) demonstrates up to 4.94× speedup.

# 5. Takeaways

  • W4A8 quantization is promising but challenging due to CUDA Core dequantization overhead.
  • Hardware-aware algorithm design (LQQ) and pipeline design (ImFP) are critical for practical speedups.
  • LiquidGEMM sets a new benchmark for high-performance LLM inference on GPUs using low-bit quantization.
  • System-level results highlight scalability and robustness, especially for large models.

# 12. NVIDIA Blackwell; The Impact of NVFP4 For LLM Inference

https://www.nota.ai/community/where-do-llms-encode-the-knowledge-to-assess-the-ambiguity-6zjbl-tmjns-l25lf

Blackwell’s 5th-generation Tensor Cores natively support FP4/FP6/FP8, allowing NVFP4 workloads to fully utilize low-precision compute and bandwidth efficiency

As batch size or input sequence length grows, the matrix-multiply load and memory pressure increase, amplifying the benefits of native FP4 execution.

A100 (Ampere) generally shows lower TPOT than RTX PRO 6000 (Blackwell) across most conditions.

This is primarily due to the A100’s higher memory bandwidth (≈2,039 GB/s via HBM2e) compared to the RTX PRO’s GDDR7 bandwidth (≈1,792 GB/s), which provides a clear advantage in a memory-bound scenario.

However, as the input length or batch size increases, the workload transitions from memory-bound to compute-bound.

In this regime, Blackwell’s native FP4/FP8 Tensor Core pathways begin to show their strength, leading to cross-over points where NVFP4-based models surpass higher-precision ones—similar to the pattern observed in the prefill phase.

# 14. Unleashing Low-Bit Inference on Asend NPUs A Comprehensive Evaluation of HiFloat Formats

# 1. Context and Motivation

  • Problem: Large Language Models (LLMs) are increasingly demanding on memory bandwidth and computational throughput. Quantization is a common approach to reduce memory overhead and accelerate inference.
  • Current Methods: Most work focuses on integer-based quantization (INT8, INT4) for weights, activations, and KV cache, but integer formats struggle with high-variance or outlier-heavy data distributions common in LLMs.
  • Emerging Solution: Low-bit floating-point formats like MXFP and NVFP4 introduce hierarchical scaling and per-tensor adjustments to maintain precision in low-bit representations. This paper evaluates HiFloat, a family of formats (HiF8 and - HiF4) specifically designed for Huawei Ascend NPUs, focusing on their ability to handle post-training quantization (PTQ) for inference.

# 2. HiFloat Formats Overview

# 2.1 HiF8 (8-bit)

Features dynamic mantissa allocation: more bits are used for small exponent values, fewer for large exponents. Components:

  • Sign bit (S) – indicates positive/negative
  • Dot field – distinguishes normal vs subnormal numbers
  • Exponent (E) – adaptive length
  • Mantissa (M) – remaining bits

Advantage: Good for wide dynamic range, capturing high-variance activations. Limitation: For static weight distributions concentrated in small ranges, HiF8 wastes bits on rare extreme values. Scaled variants (HiF8 scaled) use a per-channel scale to mitigate this.

# 2.2 HiF4 (4-bit)

  • Ultra-low bit representation uses a three-level hierarchical scaling:
  • Block of 64 elements → 8-element sub-block → 4-element micro-block.
  • Each block is scaled with E6M2 (block), 1-bit factors for sub-blocks, and 4-bit element representation (E1M2).
  • Advantage: Isolates local variance and high-magnitude outliers while preserving precision.
  • Outperforms INT4 and MXFP4 in representing both weights and activations at 4-bit.

# 3. Comparison With Other Formats

8-bit Regime:

  • INT8 excels for weight compression due to uniform representational density.
  • HiF8 and MXFP8 perform better for activations, which are dynamic and have outliers.
  • KV cache fidelity is intermediate; INT8 better for narrower ranges, HiF8/MXFP8 better for dynamic distributions. 4-bit Regime:
  • INT4 collapses due to insufficient representable levels.
  • MXFP4 improves but lacks local granularity.
  • NVFP4 and HiF4 maintain structural resilience and high SQNR across weights, activations, and KV cache.
  • HiF4 consistently preserves >96% of BF16 baseline on Qwen3-8B and 97% on openPangu-7B (Tables 4–9, Figures 6–13).

Interesting finding about precision of weights, activations, and also the difference between k and v values.

# 4. Quantization Mechanisms

# 4.1 HiF8

Determine exponent

Allocate mantissa bits based on magnitude of e:

  • e ≤ 3 → 3 bits
  • 3 < e ≤ 7 → 2 bits
  • 7 < e ≤ 15 → 1 bit
  • e > 15 → 0 bits Apply round-to-nearest (RTN) to quantize, then reconstruct value using sign and scale.

# 4.2 HiF4

Hierarchical scaling across 3 levels:

  • Level 1: Block of 64 elements → compute max → E6M2 scale S1
  • Level 2: Sub-block of 8 elements → 1-bit exponent S2
  • Level 3: Micro-block of 4 elements → 1-bit exponent S3

Normalize element:

  • Quantize to 4-bit E1M2.
  • Dequantization is simple: sum exponents and apply a bit-shift (avoiding FP multiplications).

# 4.3 MXFP / NVFP4

MXFP uses single-level block-based scaling (32-element blocks).

NVFP4 uses two-level hierarchical scaling, including per-tensor FP32 scale to prevent clipping.

Hierarchical structures are critical for 4-bit inference.

# 5. Experimental Evaluation

# 5.1 Metrics

SQNR (Signal-to-Quantization-Noise Ratio) – measures fidelity of quantized weights/activations/KV cache.

Perplexity & Accuracy – standard LLM evaluation on Wikitext-2, C4, ARC, HellaSwag, MMLU, Math500, GSM8K, LongBench.

# 5.2 Findings

W8A8 (8-bit weight/activation): INT8 is best for weights, HiF8 is competitive with floating-point advantages for activations (Figure 2–3, Tables 4–5).

W4A4 (4-bit weight/activation): INT4 fails, MXFP4 partially recovers, HiF4 and NVFP4 maintain high fidelity (Figures 6, 8–10, Table 9).

KV Cache Quantization: Key states are sparse and high-variance → HiF4/NVFP4 hierarchical scaling improves performance over uniform or single-level formats.

Long Context: HiF4 shows robustness under full 4-bit WA + KV quantization, outperforming MXFP4 and NVFP4 (Table 9, Figures 12–13).

# 5.3 Distributional Analysis

8-bit weights: concentrated distribution → INT8 uniform grid superior.

8-bit activations: heavy-tailed outliers → HiF8/MXFP8 better.

4-bit weights/activations: INT4 insufficient, HiF4 hierarchical quantization preserves the bell-shaped distribution and avoids coarse binning (Figures 11–13).

# 6. Key Takeaways

Precision vs Dynamic Range Trade-off

  • 8-bit: uniform INT8 best for static weights; floating-point better for high-variance activations.
  • 4-bit: uniform spacing fails; hierarchical scaling essential. Hierarchical Scaling
  • HiF4’s three-level approach isolates outliers while preserving local detail → robust for end-to-end 4-bit inference. Compatibility
  • HiFloat integrates well with PTQ frameworks like SmoothQuant and SVDQuant, further improving accuracy. KV Cache Sensitivity
  • High-variance Key states are critical; hierarchical scaling in HiF4/NVFP4 preserves accuracy in attention mechanisms.
  • However, V value is not so sensitive. Practical Implication
  • HiF4 is a practical, hardware-friendly low-bit format for LLM inference on Ascend NPUs, supporting long-context tasks without catastrophic loss in fidelity.

# 7. Conclusion

HiFloat formats (HiF8, HiF4) provide an effective solution for low-bit LLM inference:

  • HiF8: wide dynamic range for activations.
  • HiF4: hierarchical, ultra-low bit representation robust across weights, activations, and KV cache.
  • INT-based formats (INT4/INT8) cannot reliably handle high-variance activations or 4-bit low-bitwidth tasks. Hierarchical floating-point formats are essential for maintaining end-to-end accuracy at 4-bit precision.
