## [1] Model Parallelism on Distributed Infrastructure: A Literature Review from Theory to LLM Case-Studies

- **Inter-operator Parallelism**: The graph is partitioned into sub-graphs, where each sub-graph is assigned to a different device. This is often referred to as **pipeline parallelism**.
- **Intra-operator Parallelism**: The computation within a single operator (like a matrix multiplication) is split across devices. This is often referred to as **tensor parallelism**.

### The Core Challenges
- **Intra-operator Challenges**: The primary bottleneck is communication. To split a single operation, input tensors must be scattered across devices and output tensors must be gathered back together, requiring extremely high bandwidth.
- **Inter-operator Challenges**: The primary bottleneck is device utilization. Because the output of one partition is the input for the next, devices often sit idle waiting for data. This creates "bubbles" in the pipeline where no computation occurs.
- **Hybrid Challenges**: Because single strategies have limitations, modern implementations often use hybrid strategies combining data, inter-operator, and intra-operator parallelism. Finding the optimal combination of these strategies is a complex search problem

### The Megatron Family (NVIDIA/Microsoft)
- Strategy: Megatron models typically use intra-operator (tensor) parallelism within a single multi-GPU node (utilizing fast NVLink connections) and inter-operator (pipeline) parallelism to scale across multiple nodes151515.
- Implementation: They split the Multi-Layer Perceptron (MLP) and Self-Attention blocks over multiple GPUs16.
  - For MLPs, they split weight matrix $A$ across columns and matrix $B$ across rows. This allows them to avoid communication during the non-linear GeLU activation, requiring synchronization only at the end of the block.
- Hardware: These approaches are heavily reliant on high-speed NVLink bridges because of the high communication cost of intra-operator parallelism.

### Gopher and PaLM (Google)
- Strategy: These models run on Google's custom TPU hardware.
  - PaLM's Unique Approach: Notably, the PaLM model avoids inter-operator (pipeline) parallelism entirely. Instead, it utilizes 12-way intra-operator parallelism and massive 256-way data parallelism.
- Benefit: By avoiding pipelining, PaLM eliminates the "pipeline bubble" problem that reduces training efficiency in other models.
  
### Auto-Parallelization Frameworks
- The paper reviews automated tools designed to find optimal parallelization strategies without manual tuning.
- FlexFlow: Uses a search space called SOAP (Sample, Operator, Attribute, Parameter) to identify parallelizable dimensions23. It employs a Markov Chain Monte Carlo (MCMC) search algorithm to find efficient strategies24.
- Alpa: Uses a hierarchical approach25.Level 1: Optimizes intra-operator parallelism using integer linear programming26.Level 2: Optimizes inter-operator partitioning using dynamic programming.
- PipeDream: Focuses on pipeline parallelism by allowing multiple batches to be "in-flight" simultaneously (asynchronous scheduling) to reduce bubble size, though this can introduce memory overhead28282828.

### Conclusion and Future Directions
- The authors conclude that while specialized hardware (like NVLink and TPUs) enables massive intra-operator parallelism, the field suffers from a lack of standardization.
- Comparison Difficulty: Because every research group uses different hardware and model architectures, it is nearly impossible to objectively compare different parallelization strategies.
- Proposed Solution: The authors suggest creating a standardized benchmark dataset—similar to NAS-Bench in Neural Architecture Search—that contains a fully explored search space. This would allow researchers to evaluate new algorithms without needing access to thousands of GPUs31.

## Optimizing Distributed LLM Inference: NVLink vs. Inter-Node Networking

If you recall reading that specific types of parallelism are better suited for specific hardware connections (like NVLink vs. Ethernet), your memory is spot on.

In distributed deep learning, the "chattiness" of the parallelization strategy dictates which hardware interconnect is required. The literature distinguishes between strategies that require the high bandwidth of **NVLink** (inside a single node) versus those that can tolerate slower **inter-node connections** (across different servers).

### 1. Intra-Operator Parallelism (Tensor Parallelism)
**Best suited for: NVLink (Same Node)**

Intra-operator parallelism splits individual operations (like a single matrix multiplication) across multiple GPUs. This is often referred to as **Tensor Parallelism**.

* [cite_start]**The Bottleneck:** This method has "extreme communication requirements"[cite: 218]. [cite_start]The devices must scatter input tensors and gather output tensors for *every single layer* in the network[cite: 219].
* **Why NVLink?** Because of this intense communication frequency, standard networks (like Ethernet) are too slow. [cite_start]The high bandwidth of NVLink is required to prevent the GPUs from stalling while waiting for data[cite: 409].
* [cite_start]**Deployment:** Consequently, this strategy is almost exclusively limited to a **single compute node** where devices are connected via high-speed bridges[cite: 412].

### 2. Inter-Operator Parallelism (Pipeline Parallelism)
**Best suited for: Inter-Node Network (Across Nodes)**

Inter-operator parallelism partitions the model itself into sub-graphs (e.g., Layers 1–10 on Server A, Layers 11–20 on Server B). This is often referred to as **Pipeline Parallelism**.

* [cite_start]**The Bottleneck:** This technique has "relatively low communication requirements"[cite: 161]. Communication only occurs at the boundaries of the sub-graphs (the cut points between pipeline stages).
* [cite_start]**Why Inter-Node?** Since data transfer happens less frequently (only passing the output activations of one stage to the next), it is tolerant of the higher latency and lower bandwidth found in connections between different servers[cite: 161].

## Real-World Example: The "Megatron" Architecture
[cite_start]The "Megatron" family of models (developed by NVIDIA and Microsoft) is a classic example of a **hybrid strategy** that explicitly maps these software strategies to the hardware hierarchy[cite: 220, 500].

| Parallelism Type | Scope | Hardware Interconnect | Reason |
| :--- | :--- | :--- | :--- |
| **Intra-Operator** | Inside one server | **NVLink** | [cite_start]Needs massive bandwidth to split individual matrices[cite: 411]. |
| **Inter-Operator** | Across servers | **Ethernet / InfiniBand** | [cite_start]Lower bandwidth needs; data only moves at stage boundaries[cite: 500]. |

By using Intra-operator parallelism *within* a node to maximize GPU utilization and Inter-operator parallelism *across* nodes to scale memory, developers can train and serve models that are too large for any single device or server.
