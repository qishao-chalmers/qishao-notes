# 1. [55 2025] Scaling Long-Horizon LLM Agent via Context-Folding

The idea is simple.

For subtasks, make a branch off to another agent or the same agent, so it will conduct the task.
When the task is done, return the result and fill into the context of the main task.

<img width="1282" height="812" alt="image" src="https://github.com/user-attachments/assets/7c3b1334-31fd-41b4-b720-e5ba086063a0" />

One of the fancy thing would be for the branches, they can share some of the KV cache, since they are originated from the same main task.

Takeaway:
Use branch action and return action to manage the subtask.
Core design in the reward function:
- If the context in the main task is too long, then we can set penality to each token, so it will punish the main task from not generate new subtask.
- If the subtask task drift from the original task, first we can use another model to judge whether this is a drift, if there is, we will punish the substask from being out of scope.

# FoldGRPO Reward and Penalty Design (Generated from Dr. Chatgpt)

## 1. Overall RL Objective
- FoldGRPO optimizes the agent's trajectory, including both the main thread and branch sub-tasks.
- The agent policy is conditioned on **folded context** \(F(\tau_{<i})\), not the full interaction history.
- The learning objective combines:
  - **Task success reward** \(R_i \in \{0,1\}\) (pass/fail for the overall task)
  - **Token-level process rewards** \(Q_{i,t}\) that shape intermediate behavior.

**Token-level Advantage Calculation:**
```math
A_{b_{i,t}} = clip(R_i + Q_{i,t}, 0, 1) - mean(R_i) / std(R_i)
```
Where \(Q_{i,t}\) regularizes **branch-return behavior** at the token level.

## 2. Token-Level Process Rewards and Penalties

### (a) Unfolded Token Penalty
- **Purpose:** Prevents token-heavy operations in the main thread, which quickly exhausts context.
- **Implementation:**
  - If the main context >50% of its limit, set \(Q_{i,t} = -1\) for all main thread tokens **except branch-creating tokens**.
- **Effect:** Encourages moving heavy computations into branches.

### (b) Out-of-Scope Penalty
- **Purpose:** Keeps branch actions focused on the assigned sub-task.
- **Implementation:**
  - Use **GPT-5-nano** (This is interesting, like a judge model) to assess if branch actions deviate from the sub-task prompt.
  - If deviation detected, set \(Q_{i,t} = -0.2\) for all tokens in the branch.
- **Effect:** Maintains **scope integrity** of each branch.

### (c) Failure Penalty
- **Purpose:** Penalizes failed tool calls (e.g., unsuccessful searches or code execution).
- **Implementation:**
  - Apply \(Q_{i,t} = -1\) for tokens in a failed tool call turn.
  - Other tokens get \(Q_{i,t} = 0\).
- **Effect:** Encourages **successful and careful tool usage**.

## 3. Combined Effect
- **Main Thread:** Learns to plan **high-level strategies** without clutter.
- **Branches:** Learn to **complete sub-tasks effectively** and produce concise summaries.
- **Dense Feedback:** Reduces the problem of sparse rewards from only task-level success.
- **Outcomes:**
  - Main context compressed to ~8K tokens
  - 5–6 focused branches per task
  - Higher task success (e.g., pass@1 62% on BrowseComp-Plus)

## 4. Intuition
- Think of the reward system as a **real-time teacher:**
  - "Don't overload main context → -1"
  - "Stay on your assigned sub-task → -0.2"
  - "Failed tool call → -1"
- This shapes the agent to **naturally follow the folding workflow** without heuristics.
