# 10. Quartet: Native FP4 Training Can Be Optimal for LLMs – Summary

**Summarized by chatgpt**

## Objective
Enable **fully end-to-end 4-bit FP (MXFP4) training** of large language models (LLMs) with minimal accuracy loss, maximizing **compute and energy efficiency** on NVIDIA Blackwell GPUs.

---

## Key Contributions
1. **FP4 LLM Training Analysis**
   - First systematic study of hardware-supported **MXFP4 training**.
   - Introduces metrics:
     - **effN (parameter efficiency)** → forward-pass quantization effect.
     - **effD (data efficiency)** → backward-pass gradient bias effect.

2. **Quartet Algorithm**
   - Forward pass: **Hadamard + QuEST** → minimizes activation MSE.
   - Backward pass: **Round-to-nearest (RTN)** → maintains unbiased gradient estimates.
   - Fully low-precision GEMMs for linear layers.

3. **Scaling-Law Framework**
   - Quantizes forward and backward separately:
     ```
     L(N, D, P_forward, P_backward) = f(N * effN(P_forward), D * effD(P_backward))
     ```
   - Predicts optimal precision for **accuracy vs runtime**.

4. **Efficient GPU Implementation**
   - Fuses Hadamard transform, quantization, mask generation, and GEMM.
   - Optimized memory layout for MXFP4 scales and masks.
   - Achieves up to:
     - **2× speedup over FP8 GEMMs**
     - **1.6× end-to-end training speedup** on RTX 5090

5. **Experimental Validation**
   - Llama-family models: 30M → 7B parameters on C4 dataset.
   - Outperforms prior FP4/INT4 methods (LUQ, HALO, Jetfire, LSS) in:
     - Validation loss
     - Parameter and data efficiency
   - Stable training for large-scale LLMs.
   - Post-training MXFP4 quantization reduces perplexity vs baseline (QuaRot).

---

## Insights

### 1. Forward vs Backward Separation
- Forward pass affects **activations and parameter efficiency (effN)** → directly impacts inference speed.
- Backward pass affects **gradient correctness (effD)** → biased gradients require more data or hurt convergence.
- Separate treatment allows **best-of-both-worlds precision**: QuEST forward, RTN backward.

### 2. Gradient Bias
- Low-precision backward quantization can **systematically alter gradient magnitude/direction**.
- Causes slower convergence or requires more training data.
- Quartet addresses this with **RTN + Hadamard-based alignment**.

### 3. Practical Optimality
- MXFP4 training can be **“optimal”**:
  - Slight accuracy degradation is compensated by high throughput.
  - Full FP4 training is faster than FP8 or FP16 while maintaining comparable accuracy.

### 4. GPU-Aware Implementation
- Forward and backward GEMMs in MXFP4, with fused operations for efficiency.
- Hadamard transforms used to decorrelate inputs/gradients and reduce quantization error.
- Masking ensures low-precision clipping does not propagate bias.

---

## Takeaway
> **Quartet shows that LLMs can be trained fully in 4-bit FP precision with near-lossless accuracy if forward and backward passes are handled separately, gradient bias is managed, and GPU kernels are highly optimized. MXFP4 training is both practical and faster than higher-precision alternatives.**

## 11. :rocket: [5 2025] LiquidGEMM: Hardware-Efficient W4A8 GEMM Kernel for High-Performance LLM Serving :+1: :+1: :+1:

*This is a paper I would like to reproduce*
### 1. Introduction
This paper addresses a practical performance bottleneck in W4A8 (4-bit weights, 8-bit activations) quantization for LLMs, which is widely used for efficient inference. 

The key insight is that while W4A8 quantization theoretically reduces memory usage and improves throughput, existing W4A8 GEMM kernels underperform in practice, often being slower than W8A8 or even FP16 due to inefficient dequantization on CUDA cores before Tensor Core MMA operations.

The authors propose LiquidGEMM, a hardware-efficient W4A8 GEMM kernel that introduces two co-designed innovations:

- LiquidQuant (LQQ) – A safe, hardware-efficient quantization/dequantization algorithm.
- Implicit Fine-Grained Pipeline (ImFP) – A pipeline design that maximizes overlap between weight loading, dequantization, and MMA across GPU warps without software synchronization.

Experimental results demonstrate up to 2.90× GEMM kernel speedup and up to 4.94× end-to-end system speedup over state-of-the-art W4A8 kernels.

### 2. Background & Motivation
#### 2.1 W4A8 Quantization
Quantizes FP16/FP32 weights to 4-bit integers and activations to 8-bit integers.

Reduces memory footprint and improves throughput on Tensor Cores.

Problem: Unlike W8A8, **W4A8 requires dequantization on CUDA cores before MMA, which is costly because CUDA cores are slower than Tensor Cores.**

#### 2.2 GEMM on GPUs
GEMM (General Matrix Multiplication) is the main computational bottleneck in LLM inference.

Tensor Cores accelerate matrix multiply-accumulate (MMA) operations but only support symmetric precisions efficiently.

For asymmetric W4A8, weight dequantization is performed in the main loop, leading to a bottleneck (Figure 3, page 4).

#### 2.3 Performance Gap
Benchmarks on LLaMA2-7B and Mixtral-8×7B reveal that W4A8 is:
- Competitive at small batch sizes.
- Up to 2× slower than W8A8 at larger batch sizes.

Dequantization overhead dominates compute in large-batch and compute-bound scenarios (highlighted on pages 4–5).

Without dequantization overhead:
- W4A8 and W8A8 should be similar in compute-bound scenarios because both use INT8 MMA on Tensor Cores.
- W4A8 should even outperform W8A8 in memory-bound scenarios, since it loads half the weight data (4-bit vs 8-bit) → lower memory load (T_LD).

**Problem:**
- W4A8 requires dequantization on CUDA cores (T_DQ) before Tensor Core MMA.
- CUDA cores are much slower than Tensor Cores, and dequantization requires multiple instructions per element (especially to handle overflows).
- This adds significant overhead, which dominates the main loop at larger batch sizes.

Effect on performance curves:
- T_DQ “shifts” the expected W4A8 performance down.
- W4A8 is similar to W8A8 in memory-bound scenarios, and up to 2× slower in compute-bound cases (page 5).

Takeaway: The bottleneck isn’t the Tensor Core MMA, but CUDA-core dequantization.

- Limitation: **Tile size (M_t) is constrained by shared memory and the M_t determinnes the arithmetic instensity**.

#### 2.5 Design Implications for Efficient W4A8 GEMM

The analysis suggests two critical design principles:

Pipeline across heterogeneous units:
- Weight loading (TMA), dequantization (CUDA cores), and MMA (Tensor Cores) must be fully pipelined.
- Avoids serialization due to dequantization.

Hardware-efficient dequantization:
- The instruction cost per dequantized element (α) must be low enough to overlap with memory load or MMA.

On H100:
- Memory-bound threshold: α ≤ 5.07 instructions per element.
- Compute-bound threshold (M=150): α ≤ 5.05 instructions per element.
- Additional tasks (address calculation, index updates) further increase the CUDA core load, reinforcing the need for optimized dequantization.

### 3. LiquidGEMM Design
#### 3.1 LiquidQuant (LQQ) – Hardware-Aware Quantization
Two-level quantization:
- FP16 → INT8 (per-channel, protective range [-119, 119]).
- INT8 → UINT4 (shift into unsigned domain to avoid overflow).
Dequantization formula:
- Uses multiply-add (IMAD) and XOR instructions.
- Fully avoids overflow in UINT8 domain (Figure 8, page 8).
- Four elements dequantized with only two hardware instructions, reducing CUDA Core load.
#### 3.2 Implicit Fine-Grained Pipeline (ImFP)
Goal: overlap weight loading → dequantization → MMA efficiently.
Compare to ExCP (explicit coarse-grained pipeline):
- ExCP assigns separate warp groups for dequantization and MMA → introduces pipeline bubbles and extra memory transfers.

<img width="583" height="520" alt="image" src="https://github.com/user-attachments/assets/8783e25c-fdf6-43a0-902c-8fdc79e0dcf0" />

ImFP:
- Single-producer (Load WG) → multiple-consumer (Compute WGs).
- Each Compute WG performs both dequantization and MMA, avoiding intermediate writes to shared memory.
- Achieves implicit parallelism across warp groups (Figure 6, page 6).

**This part is to eliminate the complex of *weight loading via TMA, dequantization via CUDA Cores,and MMA via Tensor Cores.*

#### 3.3 Memory Layout Optimizations
Dual-MMA packed layout (Figure 7, page 7):
- Packs data for two consecutive MMA operations per thread.
- Aligns memory for 32 UINT4 elements per thread → reduces load instructions and shared memory conflicts.
- Improves utilization of Tensor Cores and shared memory bandwidth.

### 4. Key Contributions
- Performance Analysis: Pinpointed dequantization as the main W4A8 bottleneck.
LiquidQuant: Overflow-safe, hardware-efficient W4A8 quantization/dequantization.
- ImFP Pipeline: Implicit fine-grained pipelining maximizes GPU subsystem utilization.
- Memory Layout Optimization: Dual-MMA layout reduces load overhead and maximizes bandwidth.
System Integration: End-to-end LLM serving (LiquidServe) demonstrates up to 4.94× speedup.

### 5. Takeaways
- W4A8 quantization is promising but challenging due to CUDA Core dequantization overhead.
- Hardware-aware algorithm design (LQQ) and pipeline design (ImFP) are critical for practical speedups.
- LiquidGEMM sets a new benchmark for high-performance LLM inference on GPUs using low-bit quantization.
- System-level results highlight scalability and robustness, especially for large models.
